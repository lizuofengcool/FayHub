## 高危问题（紧急修复）

### 1. JWT中间件全局注册导致登录接口被拦截

- **位置**：`cmd/main.go` 第102行
- **具体表现**：`r.Use(internalMiddleware.JwtAuthMiddleware())` 将JWT认证作为全局中间件注册，所有请求（包括 `/api/auth/login` 和 `/api/auth/register`）都必须携带Token才能访问
- **违反规范**：架构设计原则 - 公开接口不应要求认证
- **风险等级**：高危（登录和注册接口完全不可用，系统无法正常使用）
- **当前代码**：
  ```go
  // cmd/main.go 第102行
  r.Use(internalMiddleware.JwtAuthMiddleware()) // JWT认证
  ```
- **整改方案**：移除全局JWT中间件，改为在需要认证的路由组上单独注册
  ```go
  // 删除全局注册
  // r.Use(internalMiddleware.JwtAuthMiddleware()) 
  
  // 在各路由组中按需注册（auth.go中已正确区分公开/需认证路由）
  ```

### 2. 登录时Token绑定租户ID硬编码为0

- **位置**：`internal/service/auth.go` 第66行、第105行
- **具体表现**：`Login` 和 `RefreshToken` 方法中，`GenerateToken` 的租户ID参数硬编码为0
- **违反规范**：多租户隔离铁律 - 租户ID必须从用户关联数据中获取
- **风险等级**：高危（所有用户登录后租户ID为0，多租户隔离完全失效）
- **当前代码**：
  ```go
  // auth.go 第66行
  token, err := utils.GenerateToken(user.ID, user.Username, user.Role, 0)
  // auth.go 第105行
  newToken, err := utils.GenerateToken(user.ID, user.Username, user.Role, 0)
  ```
- **整改方案**：从用户数据中获取实际租户ID
  ```go
  token, err := utils.GenerateToken(user.ID, user.Username, user.Role, user.TenantID)
  ```

### 3. 多租户隔离机制存在安全漏洞（租户ID从前端Header获取）

- **位置**：`internal/middleware/tenant.go` 第24-30行
- **具体表现**：租户ID从HTTP Header `X-Tenant-ID` 获取，客户端可随意篡改
- **违反规范**：多租户安全铁律 - 租户ID必须从JWT Token中解析，禁止从前端传参
- **风险等级**：高危（恶意用户可伪造租户ID访问其他租户数据）
- **当前代码**：
  ```go
  tenantIDStr := c.GetHeader("X-Tenant-ID")
  ```
- **整改方案**：从JWT Token的claims中获取租户ID（JWT中间件已将claims写入context）
  ```go
  tenantID, exists := c.Get("tenant_id")
  if !exists {
      c.Set("tenant_id", uint(0))
  }
  ```

### 4. 日志中间件从前端Header读取租户ID和用户ID

- **位置**：`pkg/middleware/log_context.go` 第33-40行、第96-104行、第114-118行
- **具体表现**：`LoggingMiddleware` 和 `TenantContextMiddleware`、`UserContextMiddleware` 从 `X-Tenant-ID` 和 `X-User-ID` Header读取值并注入context
- **违反规范**：多租户安全铁律 - 租户ID和用户ID禁止从前端传参
- **风险等级**：高危（恶意用户可伪造身份绕过隔离）
- **当前代码**：
  ```go
  if tenantID := c.GetHeader("X-Tenant-ID"); tenantID != "" {
      c.Set("tenant_id", tenantID)
      ctx = context.WithValue(ctx, "tenant_id", tenantID)
  }
  if userID := c.GetHeader("X-User-ID"); userID != "" {
      c.Set("user_id", userID)
      ctx = context.WithValue(ctx, "user_id", userID)
  }
  ```
- **整改方案**：从JWT claims中获取租户ID和用户ID，禁止从Header读取

### 5. 全局数据库实例暴露风险

- **位置**：`pkg/utils/db_helper.go` 第21-30行、第33-38行
- **具体表现**：存在 `GetGlobalDB()` 和 `GetInitDB()` 两个函数，均可直接绕过租户隔离
- **违反规范**：多租户隔离铁律 - 绝对禁止使用全局DB直连
- **风险等级**：高危（直接击穿租户隔离）
- **实际调用情况**：`internal/initialize/admin.go` 第16行和第78行已在使用 `GetGlobalDB()`
- **整改方案**：
  - `GetGlobalDB()` 仅保留给初始化代码使用，添加编译期或运行时访问控制
  - `GetInitDB()` 与 `GetGlobalDB()` 功能重复，应删除
  - 初始化代码（admin.go）中的调用可暂时保留，但需添加注释标记为"仅限初始化使用"

### 6. 配置文件硬编码敏感信息

- **位置**：`config.yaml` 第1-17行、`config.example.yaml` 第1-33行
- **具体表现**：数据库密码 `fayhub123`、JWT密钥 `fayhub-secret-key-2024` 硬编码在配置文件中
- **违反规范**：安全最佳实践 - 敏感信息必须从环境变量读取
- **风险等级**：高危（配置文件如被提交到版本库将导致密钥泄露）
- **整改方案**：
  ```yaml
  password: ${DB_PASSWORD}
  secret: ${JWT_SECRET}
  ```
  同时确保 `config.yaml` 已在 `.gitignore` 中（当前已配置）

### 7. main.go源码中硬编码默认密码和密钥

- **位置**：`cmd/main.go` 第22-39行
- **具体表现**：配置加载失败时，使用硬编码的默认值（密码 `fayhub123`、JWT密钥 `fayhub-secret-key-2024`）
- **违反规范**：安全最佳实践 - 源码中禁止硬编码敏感信息
- **风险等级**：高危（源码中的默认密钥可被任何人看到）
- **当前代码**：
  ```go
  cfg = &config.Config{
      Database: config.DatabaseConfig{
          Password: "fayhub123",
      },
      JWT: config.JWTConfig{
          Secret: "fayhub-secret-key-2024",
      },
  }
  ```
- **整改方案**：配置加载失败时应直接报错退出，不使用硬编码默认值

### 8. 注册接口允许前端传入tenant_id

- **位置**：`internal/service/auth.go` 第32行
- **具体表现**：`RegisterRequest` 结构体中 `TenantID` 字段标记为 `binding:"required"`，从前端请求体中获取
- **违反规范**：多租户安全铁律 - 租户ID禁止从前端传参
- **风险等级**：高危（恶意用户可注册到任意租户）
- **当前代码**：
  ```go
  type RegisterRequest struct {
      TenantID uint `json:"tenant_id" binding:"required"`
  }
  ```
- **整改方案**：租户ID应从JWT Token或邀请码中获取，不从请求体传入

### 9. HealthCheck接口未使用统一响应格式

- **位置**：`internal/controller/enter.go` 第218-230行
- **具体表现**：`HealthCheck` 方法直接使用 `c.JSON(500, gin.H{...})` 和 `c.JSON(200, gin.H{...})`，未使用 `response.Gin*` 系列方法
- **违反规范**：交付物铁律 - 所有接口必须统一API返回格式 `{code, data, msg}`
- **风险等级**：高危（响应格式不一致，前端无法统一处理）
- **当前代码**：
  ```go
  c.JSON(500, gin.H{
      "code":    500,
      "message": "系统内部错误",
      "data":    nil,
  })
  c.JSON(200, gin.H{
      "code":    200,
      "message": message,
      "data": gin.H{...},
  })
  ```
- **整改方案**：
  ```go
  response.GinError(c, errors.ErrInternalServer, "系统内部错误")
  response.GinSuccessWithMessage(c, message, gin.H{"tenant_id": tenantID, "status": "running"})
  ```

## 中危问题（需要修复）

### 10. User和Role模型未嵌入TenantModel

- **位置**：`internal/model/tenant.go` 第23行、`internal/model/rbac.go` 第7行
- **具体表现**：`User` 和 `Role` 结构体嵌入了 `BaseModel` 而非 `TenantModel`，手动添加了 `TenantID` 字段
- **违反规范**：架构设计原则 - 应使用 `TenantModel` 基类确保 `BeforeCreate` 钩子生效
- **风险等级**：中危（创建记录时租户ID可能不会被自动填充）
- **整改方案**：将 `BaseModel` 改为 `TenantModel`，移除手动定义的 `TenantID` 字段

### 11. TenantModel的BeforeCreate钩子未完成

- **位置**：`internal/model/base.go` 第24-30行
- **具体表现**：`BeforeCreate` 方法只获取了context但未完成租户ID设置逻辑
- **违反规范**：架构设计原则 - 核心功能必须完整实现
- **风险等级**：中危（功能不完整）
- **整改方案**：补充完整的租户ID自动设置逻辑

### 12. 重复Config结构体定义

- **位置**：`internal/config/config.go` 第11行、`pkg/config/config.go` 第14行
- **具体表现**：存在两个 `Config` 结构体，字段不完全一致。`internal/config` 版本只有3个字段，`pkg/config` 版本有5个字段（多了 `Logging` 和 `MultiTenant`）
- **违反规范**：工程结构铁律 - 避免重复定义
- **风险等级**：中危（配置不一致，维护困难）
- **整改方案**：统一为一个Config定义，删除 `internal/config` 中的重复定义

### 13. 重复响应工具包

- **位置**：`pkg/utils/response.go`、`pkg/response/response.go`
- **具体表现**：`pkg/utils/response.go` 定义了 `Response{Code, Message, Data}` 和 `Ok/Fail` 系列方法；`pkg/response/response.go` 定义了 `Response{Code, Message, Data, Timestamp, RequestID}` 和 `GinSuccess/GinError` 系列方法
- **违反规范**：工程结构铁律 - 避免重复实现
- **风险等级**：中危（响应结构不一致，`utils.Response` 缺少 `Timestamp` 和 `RequestID`）
- **整改方案**：删除 `pkg/utils/response.go`，统一使用 `pkg/response/response.go`

### 14. 路由注册方式不一致

- **位置**：`internal/router/auth.go` vs `internal/controller/tenant.go` 第275-308行、`internal/controller/user.go` 第358行
- **具体表现**：Auth路由在 `router/auth.go` 中注册（GVA标准模式），但Tenant和User路由在Controller文件中通过 `RegisterTenantRoutes()` 和 `RegisterUserRoutes()` 函数注册
- **违反规范**：GVA组管理模式 - 路由注册应统一在Router层
- **风险等级**：中危（架构不一致，违反单向数据流原则）
- **整改方案**：将 `RegisterTenantRoutes` 和 `RegisterUserRoutes` 移到对应的Router文件中

### 15. AuthService未遵循GVA组管理模式

- **位置**：`internal/service/auth.go`
- **具体表现**：`AuthService` 是独立结构体，没有 `var AuthServiceApp = &AuthService{}` 实例变量，而 `TenantService` 和 `UserService` 都有
- **违反规范**：GVA组管理模式 - Service应统一通过 `ServiceGroupApp` 访问
- **风险等级**：中危（架构不一致）
- **整改方案**：为 `AuthService` 添加 `var AuthServiceApp = &AuthService{}`

### 16. TenantMiddleware仅应用于System路由组

- **位置**：`internal/router/enter.go` 第38行
- **具体表现**：`TenantMiddleware()` 仅在 `/api` 的system子组上注册，Auth、Tenant、User、RBAC路由组均未应用
- **违反规范**：多租户隔离铁律 - 所有业务路由必须挂载租户中间件
- **风险等级**：中危（部分路由缺少租户上下文）
- **整改方案**：在全局或各业务路由组上统一注册 `TenantMiddleware()`

### 17. config.yaml与pkg/config Config结构不匹配

- **位置**：`config.yaml`、`pkg/config/config.go`
- **具体表现**：`config.yaml` 只有 `database/server/jwt` 三个配置段，但 `pkg/config/config.go` 的 `Config` 结构体还要求 `logging` 和 `multi_tenant` 配置段
- **违反规范**：配置管理规范 - 配置文件与代码结构必须一致
- **风险等级**：中危（配置加载可能失败或使用零值）
- **整改方案**：在 `config.yaml` 中补充 `logging` 和 `multi_tenant` 配置段

### 18. Controller中变量命名误导

- **位置**：`internal/controller/tenant.go` 第44行、第82行等多处
- **具体表现**：`CreateTenant` 和 `UpdateTenant` 的返回值命名为 `token`，但实际类型是 `*model.Tenant`
- **违反规范**：代码可读性规范 - 变量名应准确反映其含义
- **风险等级**：中危（代码可读性差，容易引起误解）
- **整改方案**：将 `token` 重命名为 `tenant`

### 19. TenantService.GetTenantList关键词搜索存在SQL注入风险

- **位置**：`internal/service/tenant.go` 第221-223行
- **具体表现**：使用 `fmt.Sprintf("%"+keyword+"%")` 拼接LIKE查询，虽然GORM的 `Where` 方法会参数化查询，但关键词中的 `%` 和 `_` 通配符未被转义
- **违反规范**：安全最佳实践 - LIKE查询应转义通配符
- **风险等级**：中危（可能导致意外的查询结果）
- **整改方案**：对关键词中的 `%` 和 `_` 进行转义

### 20. 数据库连接未配置连接池

- **位置**：`internal/initialize/database.go`
- **具体表现**：`InitDB` 函数未配置 `SetMaxIdleConns`、`SetMaxOpenConns`、`SetConnMaxLifetime`
- **违反规范**：生产环境最佳实践 - 数据库连接必须配置连接池
- **风险等级**：中危（高并发下可能导致连接耗尽）
- **整改方案**：
  ```go
  sqlDB, _ := db.DB()
  sqlDB.SetMaxIdleConns(10)
  sqlDB.SetMaxOpenConns(100)
  sqlDB.SetConnMaxLifetime(time.Hour)
  ```

## 低危问题（建议修复）

### 21. 缺少CORS中间件

- **位置**：`cmd/main.go`
- **具体表现**：未配置跨域资源共享中间件
- **风险等级**：低危（前端开发时将遇到跨域问题）
- **整改方案**：添加 `github.com/gin-contrib/cors` 中间件

### 22. 缺少请求限流中间件

- **位置**：`cmd/main.go`
- **具体表现**：未配置请求限流，登录接口可被暴力破解
- **风险等级**：低危（安全加固项）
- **整改方案**：添加 `github.com/ulule/limiter` 或类似限流中间件

### 23. 缺少优雅关机

- **位置**：`cmd/main.go` 第113行
- **具体表现**：使用 `r.Run(port)` 启动服务，不支持优雅关机
- **风险等级**：低危（部署时可能导致请求中断）
- **整改方案**：使用 `http.Server` + `signal.Notify` 实现优雅关机

### 24. 编译产物未从Git中排除

- **位置**：项目根目录存在 `fayhub.exe` 和 `main.exe`
- **具体表现**：`.gitignore` 中 `FayHub/*.exe` 路径与实际文件路径不匹配（实际路径是 `FayHub/fayhub.exe`）
- **风险等级**：低危（二进制文件不应提交到版本库）
- **整改方案**：修正 `.gitignore` 中的路径规则，并清理已提交的exe文件

### 25. checkAPIPermission简化实现返回true

- **位置**：`internal/middleware/permission.go` 第136行
- **具体表现**：`checkAPIPermission` 函数注释标注"暂时允许所有API访问"，直接返回 `true`
- **风险等级**：低危（权限控制未生效，但已有SuperAdminMiddleware作为兜底）
- **整改方案**：实现完整的API权限检查逻辑

### 26. RBACService.CreateRole未设置TenantID

- **位置**：`internal/service/rbac.go` 第240-251行
- **具体表现**：创建角色时未设置 `TenantID` 字段，租户角色将使用默认值0
- **风险等级**：低危（租户角色可能无法正确隔离）
- **整改方案**：从context中获取当前租户ID并设置到角色上
